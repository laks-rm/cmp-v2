**Data Model Overview**

The CMP data hierarchy has four levels. Understanding this is critical before building anything.

**Source** → **Clause** → **Task Template** → **Task Instance**

**Source:** The compliance obligation. e.g. \"GDPR Data Protection\" or \"AML Sanctions Screening\". Has a type (Regulation, Internal Audit, External Audit, etc.), entities in scope, effective date, and ownership.

**Clause:** A specific article, section, or checkpoint within a source. e.g. \"Article 15(1) --- Right of Access\" under GDPR. Has a clause number/ID, title, and description. One source has many clauses.

**Task Template:** The repeatable work item under a clause. Defines WHAT needs to be done, HOW OFTEN, whether review/evidence is required, and the expected outcome. One clause can have multiple task templates.

**Task Instance:** The actual executable task generated by the cron job based on the template's frequency. This is what a PIC works on, uploads evidence to, and submits for review. Generated automatically.

Example: Source \"GDPR\" → Clause \"Art.15 Right of Access\" → Task Template \"Review access request log\" (Monthly, Evidence Required) → Task Instance \"T-2401 Review access request log --- April 2026\" (due Apr 30, assigned to Sarah M.)

**Module 1A: Source Creation Wizard**

A multi-step wizard that guides the user through creating a complete source with clauses and task templates. Two entry paths: Manual and AI-assisted.

**Wizard Step 1: Source Information**

Fields collected:

-   Source Title (required) --- free text, min 5 chars

-   Source Type (required) --- dropdown: Regulation, Internal Audit, External Audit, Policy, SOP, Other

-   Category (required) --- dropdown: AML, Sanctions, Regulatory Reporting, License, Data Protection, Consumer Protection, IT Security, Governance, Other

-   Description (optional) --- rich text area

-   Entities in Scope (required) --- entity picker with grouped typeahead, multi-select, flag icons

-   Owning Department (required) --- dropdown of active departments

-   Effective From (required) --- date picker, default today

-   Effective To (optional) --- date picker, null = ongoing

-   Risk Level (optional) --- High / Medium / Low / Not Assessed

-   PIC (optional) --- user search, leave blank for department queue

-   Default Reviewer (optional) --- user search

**At the top of Step 1, there are two entry path buttons:**

-   "Manual Entry" --- user fills all fields manually, proceeds to Step 2

-   "Upload Regulation & Auto-fill with AI" --- user uploads a document, AI parses it

**AI Upload Path (Branch)**

1.  User clicks "Upload Regulation & Auto-fill with AI"

2.  File picker opens. Supported formats: PDF, DOCX, TXT. Max size: 25MB.

3.  File uploads to server. Loading spinner with "AI is analyzing your document\..."

4.  AI (Google Gemini / OpenAI) processes document and returns structured JSON:

    -   Suggested source title, type, category, description

    -   Extracted clauses: \[{clause_number, title, description}\]

    -   Suggested task templates per clause: \[{title, description, frequency, review_required, evidence_required, expected_outcome, priority}\]

5.  All AI-suggested fields are pre-filled but highlighted in purple with an "AI Suggested" badge

6.  User reviews every field --- can accept, edit, or clear each one

7.  User clicks "Next" to proceed to Step 2 with clauses and templates pre-populated

**⚠ Error Handling --- AI Upload**

  ---------------------------------- -----------------------------------------------------------------------------------------------------------------------------------------
  **Scenario**                       **System Handling**

  **Unsupported file format**        Show toast: "Only PDF, DOCX, and TXT files are supported." File picker stays open.

  **File too large (\>25MB)**        Show toast: "File exceeds 25MB limit. Please upload a smaller file."

  **AI API timeout (\>60s)**         Show error: "AI processing timed out. You can retry or continue with manual entry." Show "Retry" and "Switch to Manual" buttons.

  **AI API error (500/503)**         Show error: "AI service is temporarily unavailable. Please try again later or use manual entry." Log error for monitoring.

  **AI returns empty/garbage**       Show warning: "AI could not extract meaningful content from this document. Please check the file and retry, or enter details manually."

  **AI returns partial results**     Pre-fill what was extracted, show warning: "Some sections could not be parsed. Please review and complete manually."

  **Upload interrupted (network)**   Show error: "Upload failed. Please check your connection and try again."

  **Duplicate file name**            Allow --- files stored with unique IDs, filename is metadata only
  ---------------------------------- -----------------------------------------------------------------------------------------------------------------------------------------

**Wizard Step 2: Clauses**

An editable table where each row is a clause/article/section of the source.

**Table columns:**

-   \# --- auto-incrementing sequence number

-   Clause Number/ID --- user-entered text (e.g. Art.15(1), Rule 3.2, Section A). Required, unique within source.

-   Title --- required, min 3 chars

-   Description --- optional, expandable text area (click to expand row)

-   Tasks --- count badge showing how many task templates are defined under this clause

-   Active --- toggle switch (default on)

-   Actions --- Edit (expand row), Delete (soft), Move Up/Down, Add Tasks

**Table interactions:**

-   "+ Add Clause" button below the table --- adds a new empty row

-   Drag-and-drop reordering via handle on the left

-   Click "Add Tasks" on a clause row to expand and show the task template sub-table for that clause

-   If AI path was used: table is pre-populated. Each AI-generated row has a purple "AI" badge.

-   Bulk actions: Select All, Delete Selected, Activate/Deactivate Selected

**⚠ Error Handling --- Clauses**

  --------------------------------------- ----------------------------------------------------------------------------------------------------------------------
  **Scenario**                            **System Handling**

  **Duplicate clause number**             Inline validation error: "Clause number already exists in this source." Block save.

  **Empty clause title**                  Inline validation: "Title is required." Block save.

  **Zero clauses**                        Block proceeding to Step 3. Show: "At least one clause is required."

  **100+ clauses**                        Allow but show performance warning. Paginate table at 50 rows.

  **Delete clause with existing tasks**   If source is active with generated tasks: block delete, show "Deactivate instead." If source is draft: allow delete.

  **Reorder while tasks exist**           If source is draft: allow freely. If active: reorder is cosmetic only, does not affect existing tasks.

**Wizard Step 3: Task Templates**

For each clause, define one or more task templates. The user can expand each clause to see and edit its task templates, or use a flat view showing all templates across all clauses.

**Per-clause task template table columns:**

-   \# --- sequence within clause

-   Task Title --- required

-   Frequency --- required dropdown: Daily, Weekly, Bi-weekly, Monthly, Quarterly, Semi-annually, Annually, One-time, Ad-hoc

-   Due Offset (days) --- required number. Days after period start. e.g. 15 for monthly = due on the 15th.

-   Review Required --- toggle (Yes/No)

-   Evidence Required --- toggle (Yes/No)

-   Evidence Description --- shown if evidence required = Yes. Text describing what to upload.

-   Expected Outcome --- optional text. Guidance for PIC on what success looks like.

-   Priority --- High / Medium / Low

-   Assignment --- dropdown: Source PIC, Department Queue, Round-robin, Manual

-   Reminder --- multi-select chips: 7 days, 3 days, 1 day before due

-   Escalation --- number field: days after due date to escalate. Plus dropdown: to whom.

-   Active --- toggle

If AI path was used: templates are pre-populated under each clause with purple "AI" badges.

User can add more templates to any clause, edit existing ones, or delete/deactivate.

**⚠ Error Handling --- Task Templates**

  --------------------------------------------- -----------------------------------------------------------------------------------------------------------------------------------
  **Scenario**                                  **System Handling**

  **Clause with zero templates**                Warning (non-blocking): "Clause X has no task templates. Tasks will not be generated for this clause." Allow save.

  **Due offset \> period length**               Validation warning: "Due offset of 45 days exceeds monthly period (30 days). Task will be due after period ends." Allow but flag.

  **Evidence required but no description**      Warning: "Consider adding evidence description to guide the PIC." Non-blocking.

  **Review required but no reviewer logic**     Block save: "Review is required but no reviewer assignment logic selected."

  **Reminder days \> due offset**               Validation warning: "Reminder at 7 days before due, but task is due in 5 days after period start." Auto-cap.

  **Escalation with no escalation target**      Block: "Escalation days set but no escalation target selected."

  **Frequency = One-time for ongoing source**   Allow --- task generated once, then template auto-deactivates after generation.

  **Frequency = Ad-hoc**                        No cron generation. Tasks created manually when needed. Template serves as a template for manual creation.
  --------------------------------------------- -----------------------------------------------------------------------------------------------------------------------------------

**Wizard Step 4: Review & Confirm**

Full summary of everything the user is about to create:

-   Source details: title, type, category, entities, department, dates, PIC, risk level

-   Clause summary: count of clauses, list with clause numbers and titles

-   Task template summary: total count, breakdown by frequency, review required count, evidence required count

-   Warnings/validations: any non-blocking warnings from Steps 2 and 3 shown here

-   Status selection: Draft / Active / Pending Assignment

**Actions:**

-   "Back" --- return to previous step (state preserved)

-   "Save as Draft" --- save everything with status = Draft. No tasks generated yet.

-   "Create & Activate" --- save with status = Active. Cron job will start generating tasks from next cycle.

  ----------------------------------------------------------- -----------------------------------------------------------------------------------------------------------------------------

**Source Versioning**

When a material change is made to an active source (entity scope reduction, structural task template changes, monitoring logic overhaul), the system creates a new version:

8.  User makes the change

9.  System detects it's a versioning-required change

10. Dialog: "This change requires a new version. Current version will be superseded. Continue?"

11. User confirms → system increments version_number on source

12. All clauses and templates are cloned to the new version

13. User's change is applied to the new version

14. Old version's tasks continue as-is. New tasks generated from new version.

15. Audit log: source_version_created, old_version_superseded

**⚠ Error Handling --- Editing**

  ------------------------------------------------------ ------------------------------------------------------------------------------------------------------------------------------------------------------------------------
  **Scenario**                                           **System Handling**

  **Edit conflict (two users editing simultaneously)**   Optimistic locking: show "This source was modified by \[user\] at \[time\]. Your changes:" with Overwrite / Reload options.

  **Deactivate clause with in-progress tasks**           Allow deactivation. In-progress tasks continue. Show confirmation: "3 tasks are in progress under this clause. They will continue but no new tasks will be generated."

  **PIC reassignment on active source**                  Show count of affected tasks: "12 unstarted tasks will be reassigned to \[new PIC\]. 5 in-progress tasks will remain with \[old PIC\]. Continue?"

  **Department change on active source**                 Admin-only. Show: "All unassigned tasks will move to new department queue. Assigned tasks stay with current PIC."

  **Attempt to edit archived source**                    Block all edits. Show: "This source is archived. Unarchive to edit."

  **Attempt to delete active source**                    Block. Show: "Active sources cannot be deleted. Set to Inactive or Archive instead."
  ------------------------------------------------------ ------------------------------------------------------------------------------------------------------------------------------------------------------------------------

**Module 2: Task Generation Engine (Cron Job)**

The automated system that creates task instances from active task templates based on their frequency schedules.

**2.1 Cron Job Specification**

  --------------- ---------------------------------- ---------------------------------------------------------------------------------------------------------------------------------
  **Parameter**   **Value**                          **Notes**

  Job name        generate_tasks                     

  Schedule        Daily at 00:05 UTC                 Runs once daily. Generates all tasks due in the upcoming period.

  Fallback        Manual trigger via Admin Console   If cron misses, admin can trigger manually

  Idempotency     Yes                                If run twice for same day, does not create duplicate tasks. Check: exists task with same template_id + period_start + entity_id

  Locking         Advisory lock (DB level)           Prevents concurrent execution

  Timeout         10 minutes max                     Alert if exceeds

  Logging         Structured JSON to audit log       task_generation_completed with counts
  --------------- ---------------------------------- ---------------------------------------------------------------------------------------------------------------------------------

**2.2 Generation Logic (Per Template)**

16. Query all task templates where: is_active = true AND parent clause is_active = true AND parent source status = Active

17. For each template, determine if a task needs to be generated today:

    -   Daily: generate every day

    -   Weekly: generate on the configured day_of_week (default Monday)

    -   Bi-weekly: generate every other configured day_of_week

    -   Monthly: generate on day 1 of each month (or configured day)

    -   Quarterly: generate on day 1 of quarter months (Jan, Apr, Jul, Oct or configured)

    -   Semi-annually: generate on day 1 of configured months

    -   Annually: generate on configured date each year

    -   One-time: generate once, then auto-deactivate template

    -   Ad-hoc: never auto-generated. Manual creation only.

18. For each entity in the source's entities_in_scope:

    -   Check idempotency: does a task already exist for this template + period + entity?

    -   If no: create task instance

19. Set task fields from template: title, description, priority, review_required, evidence_required, expected_outcome

20. Calculate due_date = period_start + due_date_offset_days

21. Assign PIC based on assignment_logic (Fixed → source PIC, Dept Queue → null, Round-robin → next in rotation)

22. Assign reviewer based on reviewer_logic

23. Set status = Not Started, assignment_status = Assigned or Unassigned

24. Write audit log entry: task_created

**2.3 Reminder & Escalation Cron Jobs**

**Reminder Job**

  --------------- --------------------------------------------------------------------------------------------------------------------------------------------------------------------
  **Parameter**   **Value**

  Job name        send_reminders

  Schedule        Daily at 08:00 UTC

  Logic           For each task where status IN (Not Started, In Progress): check if today is (due_date - reminder_days_before\[n\]) and reminder not already sent for this interval

  Action          Send Slack DM to PIC + in-app notification. Record reminder_sent_dates.

  Channels        Slack DM, In-app notification, Email (configurable per user)
  --------------- --------------------------------------------------------------------------------------------------------------------------------------------------------------------

**Overdue Check Job**

  --------------- -----------------------------------------------------------------------------------------------------------------------------
  **Parameter**   **Value**

  Job name        check_overdue

  Schedule        Daily at 00:30 UTC

  Logic           For each task where due_date \< today AND status IN (Not Started, In Progress): set status = Overdue if not already Overdue

  Action          Update status, set overdue_flagged_at, write audit log: task_overdue_flagged

  Notification    Slack message to #compliance-alerts. DM to PIC.
  --------------- -----------------------------------------------------------------------------------------------------------------------------

**Escalation Job**

  --------------- ---------------------------------------------------------------------------------------------------------------------
  **Parameter**   **Value**

  Job name        check_escalations

  Schedule        Daily at 09:00 UTC

  Logic           For each overdue task where (today - due_date) \>= escalation_days_after AND escalation not already sent

  Action          Send escalation notification to escalation_to target. Set assignment_status = Escalated. Record escalation_sent_at.

  Notification    Slack DM to dept manager/CMP manager. In-app alert. Email.

  Audit           escalation_triggered with task_id, escalated_to, days_overdue
  --------------- ---------------------------------------------------------------------------------------------------------------------

**⚠ Error Handling --- Cron Jobs**

  ----------------------------------------------------- -----------------------------------------------------------------------------------------------------------------------------------------
  **Scenario**                                          **System Handling**

  **Cron job fails to run**                             Health check monitors cron. If generate_tasks hasn't run by 01:00 UTC, alert admin via Slack and email.

  **Database unavailable during generation**            Job retries 3 times with exponential backoff (30s, 2min, 10min). If all fail, alert admin. No partial state --- transaction per entity.

  **Task generation creates \>1000 tasks in one run**   Allow but log warning. This happens with large source scopes. Consider batching.

  **Duplicate detection fails (race condition)**        Unique constraint on (template_id, period_start, entity_id) in database prevents duplicates at DB level.

  **Template deactivated mid-generation**               Check is_active at generation time. Deactivated templates are skipped.

  **Source marked Inactive during generation**          Check source status at generation time. Inactive sources are skipped.

  **Reminder for unassigned task**                      Send to department channel instead of PIC DM. Include "Assign to me" button.

  **Escalation target user is inactive**                Escalate to next level (CMP Manager). If no one available, alert Super Admin.

  **Slack API failure during notification**             Queue for retry. Max 3 retries. If all fail, fall back to email and in-app notification. Log failure.
  ----------------------------------------------------- -----------------------------------------------------------------------------------------------------------------------------------------

**Module 3: Task Lifecycle & Execution**

How tasks flow from generation to completion, including all branches and edge cases.

**3.1 Task Status Transitions**

  --------------------- ---------------- -------------------------------------------- -----------------
  **From**              **To**           **Trigger**                                  **Who Can**

  (generated)           Not Started      Cron job creates task                        System

  Not Started           In Progress      PIC opens and starts working                 PIC

  Not Started           Overdue          Due date passed without action               System (cron)

  In Progress           Submitted        PIC clicks Submit                            PIC

  In Progress           Overdue          Due date passed                              System (cron)

  Overdue               In Progress      PIC starts working on overdue task           PIC

  Overdue               Submitted        PIC submits overdue task                     PIC

  Submitted             Pending Review   Auto-transition if review_required = true    System

  Submitted             Approved         Auto-transition if review_required = false   System

  Pending Review        Approved         Reviewer approves                            Reviewer

  Pending Review        Returned         Reviewer returns for rework                  Reviewer

  Pending Review        Rejected         Reviewer rejects                             Reviewer

  Returned              In Progress      PIC starts rework                            PIC

  Returned              Submitted        PIC resubmits after rework                   PIC

  Approved              Closed           Admin closes or auto-close                   System/Admin

  Rejected              Closed           Auto-close on rejection                      System

  Any (except Closed)   Closed           Admin force-close with reason                Admin
  --------------------- ---------------- -------------------------------------------- -----------------

**3.2 Task Detail Page**

Shown when user clicks a task in the tracker:

-   Header: task code (T-NNNNN), title, status badge, priority, due date, overdue indicator if applicable

-   Context bar: source name + code, clause reference, entity, department

-   Ownership: PIC (with Reassign button), Reviewer, Assignment status

-   Instructions: description from template, expected outcome, evidence description

-   Evidence section: drag-and-drop upload area. List of uploaded files with name, size, uploader, timestamp, download link, delete button (PIC only, before approval)

-   Actual Outcome: text field for PIC to describe what they found/did

-   Action bar (Reviewer): Approve, Return with Comment, Reject with Reason

-   Audit trail: inline log of every change to this specific task

**3.3 Evidence Upload**

-   Supported formats: PDF, DOCX, XLSX, CSV, PNG, JPG, MSG, EML, ZIP

-   Max file size: 25MB per file, 100MB total per task

-   Max files: 20 per task

-   Storage: cloud object storage (GCS/S3) with signed URLs

-   Virus scan: on upload (async). If malware detected, file removed + user notified.

-   Evidence status auto-calculated: Not Required (evidence_required=false), Missing (required + zero files), Partial (required + files exist but \< expected), Complete (heuristic or manual confirm)

**⚠ Error Handling --- Task Execution**

  ------------------------------------------------ ---------------------------------------------------------------------------------------------------------------------------------
  **Scenario**                                     **System Handling**

  **Submit without evidence when required**        Block submission: "Evidence is required. Please upload at least one file."

  **Submit with zero content in actual_outcome**   Allow --- actual_outcome is optional. Reviewer can return if insufficient.

  **Reviewer not assigned when task submitted**    If reviewer_logic = Dept Manager: auto-assign. If no manager: escalate to CMP Manager. If still none: alert Admin.

  **PIC tries to edit approved task**              Block: "This task has been approved and is locked."

  **PIC deactivated while task in progress**       Task enters "Unassigned" state. Notification to dept manager. Task appears in dept queue.

  **Reviewer deactivated while review pending**    Auto-reassign to backup reviewer. If none: dept manager. Notification sent.

  **Multiple return cycles (\>3)**                 Allow but show warning to reviewer: "This task has been returned 3+ times. Consider escalating."

  **Task overdue then completed**                  Task closes as Approved but overdue flag remains in reporting. Audit shows was_overdue = true.

  **Evidence upload fails (network)**              Retry with resume. Show: "Upload failed. Retrying\..." If 3 retries fail: "Upload failed. Please try again." File not attached.

  **Evidence file corrupted**                      Virus scan / integrity check. If fails: "File appears corrupted. Please re-upload."

  **Concurrent submit (two browser tabs)**         First submit wins. Second sees: "This task was already submitted." Refresh to see current state.

  **Comment contains XSS/injection**               Sanitize all user input. Strip HTML tags. Escape special characters.
  ------------------------------------------------ ---------------------------------------------------------------------------------------------------------------------------------

**Module 4: Review Management**

Quality control layer. Card-based queue for deliberate review decisions.

**4.1 Review Queue UX**

Cards not table rows. Each card shows: task ID, source version, clause ref, priority, status, title, entity, submitter name, submission time, evidence count, comment count, SLA countdown, action buttons (Approve/Return/Reject).

**4.2 Review SLA**

-   Configurable per source or globally (default: 3 business days)

-   SLA countdown shown on each review card: "2d remaining" / "SLA breached 1d ago"

-   SLA breach triggers notification to reviewer and their manager

-   SLA breach count tracked per reviewer for workload reporting

**4.3 Review Filters**

-   Quick chips: Pending Review, Returned, SLA Breach, Recently Approved

-   Advanced: reviewer, source, entity, priority, department, evidence status, submission date

**⚠ Error Handling --- Review**

  --------------------------------------------------- --------------------------------------------------------------------------------------------------------
  **Scenario**                                        **System Handling**

  **Approve without viewing evidence**                Allow but log: reviewer_approved_without_evidence_view = true. Configurable: can be blocked by policy.

  **Reject without reason**                           Block: "Rejection reason is required."

  **Return without comment**                          Block: "Please explain what needs to be fixed."

  **Reviewer tries to review their own submission**   Block: "You cannot review a task you submitted." (self-review prevention, configurable)

  **Review action on already-reviewed task**          Show: "This task was already reviewed by \[user\] at \[time\]." Refresh.

  **SLA calculation across weekends/holidays**        Configurable: count business days only (requires holiday calendar) or calendar days.

**⚠ Error Handling --- Global**

  ------------------------------------------- --------------------------------------------------------------------------------------------------
  **Scenario**                                **System Handling**

  **Login with wrong password**               Show: "Invalid email or password." After 5 failures: lock for 15 min.

  **JWT expired mid-session**                 Attempt silent refresh. If refresh fails: redirect to login. Save current URL for return.

  **Password reset for non-existent email**   Show same message: "If this email exists, a reset link was sent." (prevent enumeration)

  **Search returns \>500 results**            Paginate. Show first 20. "Showing 20 of 547 results. Refine your search."
